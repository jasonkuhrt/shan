## The skill mutation loop

Most skill mutations use the same high-level machine. `skills on`, `skills off`, `skills move`, `doctor` repairs, and history restore all fit this shape directly. Install flows are close, but they may do extra bootstrap filesystem work before entering the managed graph loop:

```text
requested command
  -> resolve targets in the initial scope
  -> expand namespace targets into concrete skills
  -> resolve dependency and dependent relationships across scopes
  -> compute the exact final graph implied by the flags
  -> validate declarations, cycles, scope legality, and final graph validity
  -> plan filesystem work (symlinks, library moves, routers, mirrors, history)
  -> apply the mutation
  -> reconcile mirror agents
  -> record snapshot/history
  -> print a full ASCII report of what was selected, cascaded, blocked, skipped, and changed
```

That shared loop is why shan feels like one system instead of a pile of unrelated commands. The vocabulary and invariants stay the same even when the mutation direction changes.

## Skill graph model

The skills subsystem is a dependency-aware graph manager layered on top of Claude Code's filesystem conventions.

### Nodes

- **core skills** are real outfit directories that stay active and participate in validation
- **pluggable skills** are symlinks from the outfit into the library
- **namespace roots** may be selectable and may also be dependency targets
- **shadowed names** may exist in both project and user libraries; shan resolves them deterministically by scope rules

### Edges

- edges come from `dependencies: [...]` in `SKILL.md` frontmatter
- a dependency may point at a standalone skill or a namespace root
- namespace dependencies are dynamic at command time: the target means all current descendants of that namespace root
- reaching inside a namespace is illegal when the namespace root itself is the dependency target
- user-scope skills cannot depend on project-only skills
- self-dependencies and cycles are illegal

### Invariants

- the active graph must stay gap-free after every command
- core and pluggable nodes both count when validating the active graph
- shan does not remember why a skill became active; it acts on the current graph, not provenance
- cross-scope mutation is allowed only when the validated graph requires it

## How each skill command bends the graph

### `skills on`

- resolves the target set
- expands any namespace targets
- computes the full missing dependency closure
- activates that closure by default
- aborts instead if you pass `--fail-on-missing-dependencies`

Use it when you know what capability you want and want shan to make the graph legal for you.

### `skills off`

- starts from the selected skills, or every pluggable skill in scope during reset
- cascades upward through active dependents by default
- leaves dependencies alone by default
- optionally removes dependencies too when you pass `--cascade-dependencies`
- refuses dependent cascade instead when you pass `--fail-on-dependents`

Use it when you want to remove capability while keeping the remaining graph valid.

### `skills move`

- preserves meaning while changing location
- fails by default if the move would strand dependencies or violate scope rules
- expands downward through dependencies only when you pass `--cascade-dependencies`
- never auto-moves dependents

Use it when you want the same graph semantics in a different scope or commitment shape.

### `doctor skills`

- audits declaration legality, reach-in mistakes, cycles, missing dependencies, mirror issues, state drift, router drift, and more
- auto-fixes only what is mechanically repairable, such as activating a missing legal dependency closure or reconciling mirror drift
- leaves invalid declarations as findings for a human to fix

Use it when the filesystem may have drifted or when namespace/dependency evolution has made yesterday's outfit invalid today.

### `skills undo` / `skills redo`

- restore prior snapshots of the managed state
- replay or reverse whole operations, not partial substeps
- do not reconstruct provenance; they restore filesystem state

Use them when the previous mutation was coherent but not the one you want anymore.

## Packed command-to-state reference

```text
Read-only path
  transcript print/dump/analyze -> interpret existing transcript files
  task dump/open                -> interpret existing task files
  skills list                   -> interpret current outfit/library/state/config

Mutation path
  skills on
    -> expand targets
    -> add missing dependency closure
    -> reconcile routers/mirrors
    -> snapshot + report

  skills off
    -> remove targets
    -> cascade through dependents by default
    -> optionally cascade through dependencies
    -> reconcile routers/mirrors
    -> snapshot + report

  skills move
    -> validate destination legality
    -> optionally expand through dependencies
    -> move library/outfit representation
    -> reconcile routers/mirrors
    -> snapshot + report

  doctor skills
    -> scan declarations + live filesystem + graph state
    -> optionally apply safe repairs
    -> print findings/fixes

  undo / redo
    -> restore prior snapshot
    -> reconcile mirrors
    -> move history pointer
```

## Why this page exists

Shan is easy to underestimate if you only read the command reference. The public value is not just "some CLI commands"; it is a typed, graph-aware control surface over Claude Code artifacts:

- transcripts become readable and analyzable
- tasks become editor-friendly and project-local
- skills become a managed dependency graph with history, repair, and cross-agent mirroring

If you are deciding whether shan is useful for you, the shortest honest answer is this:

Shan is most valuable when you want Claude Code artifacts to behave like a real system instead of a loose pile of files.
