## Global options

| Flag             | Description                                                |
| ---------------- | ---------------------------------------------------------- |
| `--all`          | Show all sessions/task lists, not just the current project |
| `--no-fix`       | Report doctor findings without applying fixes              |
| `--scope user`   | Operate on user-level outfit/library (default: project)    |
| `--global`       | Alias for `--scope user`                                   |
| `--skill <name>` | Select specific skills for `skills install` (repeatable)   |
| `--strict`       | Treat no-ops (already on, already off) as errors           |

## Skills and doctor config

Shan reads skills configuration from `~/.config/shan/config.json`.

```json
{
  "version": 1,
  "skills": {
    "defaultScope": "project",
    "historyLimit": 50,
    "agents": "auto"
  },
  "doctor": {
    "disabled": ["skills/agent-mirror"]
  }
}
```

| Key                   | Values                          | Description                                              |
| --------------------- | ------------------------------- | -------------------------------------------------------- |
| `skills.defaultScope` | `project`, `user`               | Default scope for skills commands that support scoping   |
| `skills.historyLimit` | integer                         | Max retained history entries per scope                   |
| `skills.agents`       | `"auto"`, `["claude", "codex"]` | Enabled agent views for outfit mirroring                 |
| `doctor.disabled`     | namespaced rule IDs             | Skip selected doctor rules such as `skills/agent-mirror` |

`skills.agents: "auto"` probes `claude` and `codex` on `PATH`, then caches the detected set in `~/.local/shan/cache.json` for 24 hours. Delete that cache file to force an immediate refresh.

Claude is the canonical agent for now. Any non-Claude enabled agents are managed as whole-directory mirrors of the matching Claude outfit directories.

## Doctor namespace

## Transcript namespace

### `transcript print [target]`

Render a session transcript as a readable conversation log. User messages, Claude's prose, and tool calls summarized as one-liners.

**Output**: `.claude/transcripts/<session-id>.print.md`

| Flag     | Values                                         | Description                         |
| -------- | ---------------------------------------------- | ----------------------------------- |
| `--show` | `results`, `diffs`, `thinking`, `trace`, `all` | Add detail layers (comma-separated) |

`--show` layers are additive:

- `results` — tool output bodies
- `diffs` — edit diffs shown inline
- `thinking` — Claude's thinking/reasoning blocks
- `trace` — progress and system entries
- `all` — every layer

```sh
shan transcript print
shan transcript print dc8ffe42
shan transcript print --show diffs,results
shan transcript print --show all
```

### `transcript dump [target]`

Transform JSONL transcript into navigable Markdown with columnar H1 headings.

**Output**: `.claude/transcripts/<session-id>.transcript.md`

| Flag    | Description                                |
| ------- | ------------------------------------------ |
| `--raw` | Copy original JSONL without transformation |

Headings use fixed-width columns padded with underscores for editor outline alignment:

```
# `001` `01/25/2026 22:45` `user_________________` `i noticed there is a dangerous`
```

| Column  | Width    | Content                       |
| ------- | -------- | ----------------------------- |
| n       | 3 chars  | Entry number, zero-padded     |
| time    | 16 chars | Timestamp as mm/dd/yyyy hh:mm |
| type    | 21 chars | Entry type                    |
| summary | 30 chars | Type-specific summary         |

Summary extraction varies by type: user messages show first 30 chars, assistant entries show tool names with counts (`Bash ×2, Read`), summaries show `(compaction)`, and so on.

```sh
shan transcript dump
shan transcript dump dc8ffe42
shan transcript dump --raw
```

### `transcript analyze [target]`

Produce a terminal visualization of context window consumption and request patterns.

**Output**: `.claude/transcripts/<session-id>.viz.txt`

The visualization contains:

1. **Elapsed time chart** — cumulative time from session start, using `▁▂▃▄▅▆▇█` block characters
2. **Token chart** — cumulative token consumption with the same block precision
3. **Dimension tracks** — one row per dimension, one character per entry:

| Track | Symbols                                                        |
| ----- | -------------------------------------------------------------- |
| type  | `◦` user, `●` assistant, _(space)_ progress/system/snapshot    |
| skill | `◆` initial load, `╰` progressive @ref, `◇` other              |
| tool  | `▢` Bash, `▤` Read, `▣` WebFetch, `▥` Grep, `▦` Edit, `▧` Glob |
| cache | `●` hit, `○` miss                                              |
| model | `◈` opus, `◇` sonnet, `◦` haiku                                |
| files | `1`-`9` count, `+` for 10+                                     |
| trunc | `†` truncated                                                  |
| error | `×` failed                                                     |
| alert | `‼` token delta > 5000                                         |
| top   | `1`-`9` rank in top consumers                                  |

4. **Top consumers** — the 9 assistant turns with largest token deltas, showing rank, turn index, delta, tools, skill, model, and cache status
5. **Summary** — entry count, cache hit rate, errors, truncations, alerts

```sh
shan transcript analyze
shan transcript analyze dc8ffe42
```

## Skills namespace

### `skills` / `s` / `skills list`

Show the effective outfit across all layers: core skills, pluggable-on, pluggable-off, budget, the active dependency graph, and any configured mirror agents.

```sh
shan skills
shan s
shan skills list
```

Agent resolution is configured in `~/.config/shan/config.json`:

```json
{
  "version": 1,
  "skills": {
    "agents": "auto"
  }
}
```

`skills list` reports these sections when present:

- `Core (user)` / `Core (project)` — real directories shan does not manage
- `On (user)` / `On (project)` — pluggable skills currently equipped from the library, with `own=<n>` and `deps=<n>` cost columns
- `Off` — available pluggable skills not currently equipped
- `Budget` — approximate character usage against the model-visible skills budget
- `Dependency graph` — ASCII rendering of the active dependency graph with shared-node markers
- `Agent mirrors` — configured mirror directories such as Codex

Use an explicit array such as `["claude", "codex"]` to pin the enabled set instead of relying on `"auto"`.

### `skills on <targets> [--scope user] [--strict] [--fail-on-missing-dependencies]`

Create outfit symlinks for the specified skills or groups. Targets are comma-separated colon-syntax names.

A group name includes all descendant leaf skills. Already-on skills are reported as skips; pass `--strict` to treat them as errors.

By default, `skills on` activates the full missing dependency closure before mutating anything. Pass `--fail-on-missing-dependencies` to refuse instead; shan prints the exact dependency closure that would be activated and the exact rerun command with explicit targets.

```sh
shan skills on playwright
shan skills on ts:tooling,linear,playwright
shan skills on ts
shan skills on --scope user gel
shan skills on dispatch --fail-on-missing-dependencies
```

### `skills off [targets] [--scope user] [--strict] [--cascade-dependencies] [--fail-on-dependents]`

Remove outfit symlinks. No targets means reset — turn off every pluggable skill in the scope while still preserving a valid dependency graph.

Two graph flags control the extra closure:

- default: cascade through active dependents
- `--fail-on-dependents`: refuse instead of cascading through dependents
- default: leave dependencies alone
- `--cascade-dependencies`: also remove the dependency closure of the selected skills

These directions are independent. Shan validates the final remaining graph before mutating.

```sh
shan skills off playwright
shan skills off ts
shan skills off cmux --fail-on-dependents
shan skills off dispatch --cascade-dependencies
shan skills off                                # reset all pluggable
shan skills off --scope user
```

### `skills move <axis> <direction> <targets> [--scope user] [--strict] [--cascade-dependencies]`

Migrate skills between scopes or commitments. Atomic batch — all targets validate before any move executes.

**Axes and directions**:

| Command                | Effect                                             |
| ---------------------- | -------------------------------------------------- |
| `move scope up`        | project library → user library                     |
| `move scope down`      | user library → project library                     |
| `move commitment up`   | pluggable → core (copy to outfit, stop managing)   |
| `move commitment down` | core → pluggable (move to library, create symlink) |

Scope and commitment cannot change in one invocation.

Moves are dependency-aware. If the requested move would break the graph, shan aborts before mutating anything. Pass `--cascade-dependencies` to expand the move set downward through the dependency closure. Dependents are never auto-moved.

```sh
shan skills move scope up playwright
shan skills move scope down ts:tooling
shan skills move scope up dispatch --cascade-dependencies
shan skills move commitment up playwright
shan skills move commitment down playwright
shan skills move scope down tips,linear
```

### `doctor skills [--no-fix]`

Run 18 diagnostic aspects against the outfit via the root doctor command. Default: detect and auto-fix. Use `--no-fix` for report-only mode.

**Aspects**: agent-mirror, broken-symlink (with git rename detection), state-drift, new-leaf, stale-router, orphaned-router, orphaned-scope, stale-gitignore, corrupt-library-entry, frontmatter-mismatch, name-conflict, duplicate-name, shadow, stale-shadow, cross-scope-install, dependency-declaration, dependency-cycle, dependency-active-graph.

The `agent-mirror` aspect is intentionally defensive around janky layouts. When Codex is enabled, doctor will:

- create a missing `.codex/skills` or `~/.codex/skills` mirror
- accept equivalent symlinks that resolve to the canonical Claude outfit through a different path
- migrate a real Codex-owned skills directory into Claude when the Claude outfit is missing
- merge non-conflicting real Codex and Claude directories before restoring the symlink
- stop on same-name content conflicts instead of guessing

Auto-fix mode applies every fixable finding except explicit conflict/manual-review cases.

```sh
shan doctor skills
shan doctor skills --no-fix
```

### `skills migrate [--execute]`

Migrate from the legacy flat skill inventory (`~/.claude/skill-inventory/`) to the hierarchical library structure (`~/.claude/skills-library/`). Preview mode by default; pass `--execute` to apply.

```sh
shan skills migrate
shan skills migrate --execute
```

### `skills install <source> [--skill <name>]... [--all] [--scope user]`

Import skills from a `skills.sh` source into shan's own library/outfit model.

This command is intentionally non-interactive: pass one or more `--skill` flags or `--all`.

Behavior:

- shells out to `npx skills add <source> --agent claude-code --copy`
- installs into the canonical Claude outfit first
- normalizes imported skill names into shan's layout
- converts the imported skills into pluggable library entries
- generates namespace routers when needed
- reconciles any configured mirror agents afterward by re-symlinking whole agent skills directories

```sh
shan skills install vercel-labs/agent-skills --skill typed-api-dx-review
shan skills install vercel-labs/agent-skills --skill playwright --skill linear
shan skills install vercel-labs/agent-skills --all --scope user
```

### `skills install-user`

Install shan's bundled user-level skills into `~/.claude/skills-library/`, equip them, and reconcile any configured mirror agents.

```sh
shan skills install-user
```
