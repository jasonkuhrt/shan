If you want the full mental model for how shan evaluates and mutates the skill system, read [Shan state machine](../concepts/state-machine) first. This guide focuses on the hands-on command flows.

## See what you have

```sh
shan skills list
```

The output groups skills by scope and status:

- **Core (user)** / **Core (project)** — real directories shan does not manage
- **Pluggable on (user)** / **Pluggable on (project)** — symlinks from outfit to library, with both own cost and dependency-closure cost
- **Pluggable off** — in the library but not currently equipped
- **Budget** — total approximate character cost of all model-visible skills against Claude Code's character budget
- **Dependency graph** — ASCII visualization of the active dependency graph, including shared nodes
- **Agent mirrors** — optional mirror directories shan keeps in sync with the canonical Claude outfit

## Mirror active skills to other agents

Shan keeps Claude Code as the canonical outfit:

- `~/.claude/skills/`
- `.claude/skills/`

Other agents can be configured as whole-dir symlink mirrors of that active set. Today the built-in agent names are `claude` and `codex`, with Claude remaining canonical.

Default config:

```json
{
  "version": 1,
  "skills": {
    "agents": "auto"
  }
}
```

Put that in `~/.config/shan/config.json`.

`"auto"` probes `claude` and `codex` on `PATH`, then caches the detected set in `~/.local/shan/cache.json` for 24 hours.

If you want to force a fresh auto-detection immediately, delete `~/.local/shan/cache.json`.

To pin the enabled set explicitly:

```json
{
  "version": 1,
  "skills": {
    "agents": ["claude", "codex"]
  }
}
```

When the resolved agent set includes `codex`, every shan skill mutation reconciles these generated views:

- `~/.codex/skills/`
- `.codex/skills/`

Mirrored directories are fully owned by shan:

- `~/.codex/skills/` becomes a symlink to `~/.claude/skills/`
- `.codex/skills/` becomes a symlink to `.claude/skills/`

If shan finds a real Codex skills directory instead of the expected symlink, it migrates non-conflicting entries into the canonical Claude outfit and then replaces the Codex directory with the symlink. If both agents define the same skill name differently, shan stops and reports a conflict instead of guessing.

Equivalent symlink paths are accepted too. If a Codex mirror already resolves to the canonical Claude outfit through a different path string, shan treats that layout as healthy instead of rewriting it unnecessarily.

## Turn skills on

```sh
shan skills on playwright
shan skills on ts:tooling,linear,playwright
shan skills on ts                              # whole group
shan skills on --scope user gel                # user-level
shan skills on dispatch --fail-on-missing-dependencies
```

Targets use colon-syntax. A group name (like `ts`) includes all descendant leaf skills. Comma-separated for batches.

If a skill is already on, it's reported as a skip (not an error). Pass `--strict` to treat skips as errors.

Dependencies are first-class. By default, `skills on` activates the full missing dependency closure before mutating the outfit. If you want a dry refusal instead of automatic activation, pass `--fail-on-missing-dependencies`; shan prints the exact dependency closure and the exact rerun command with explicit targets.

Dependency declarations use frontmatter:

```yaml
---
name: dispatch
description: Dispatch work to parallel Claude Code sessions.
dependencies:
  - cmux
---
```

A dependency may target a standalone skill or a namespace root such as `foo`. Reaching inside a namespace root is invalid: if `foo` is the dependency target, `foo:bar` is not a legal substitute.

## Turn skills off

```sh
shan skills off playwright
shan skills off ts                             # whole group
shan skills off cmux --fail-on-dependents
shan skills off dispatch --cascade-dependencies
shan skills off                                # reset: turn off ALL pluggable skills
shan skills off --scope user                   # reset at user scope
```

No targets means reset — shan starts by selecting every pluggable skill in the scope, then may cascade through dependents in other scopes too if that is required to preserve a valid remaining graph.

`skills off` has two independent graph flags:

- Default behavior cascades through active dependents. Pass `--fail-on-dependents` to refuse instead.
- Dependency removal is opt-in. Pass `--cascade-dependencies` to also remove the dependency closure of the selected skills.

Both rules are validated against the final remaining graph. If the requested flags would leave the graph invalid, shan aborts before mutating anything.

## Move skills between scopes

Move a skill from the project library to the user library (or vice versa):

```sh
shan skills move scope up playwright           # project → user
shan skills move scope down ts:tooling         # user → project
shan skills move scope up dispatch --cascade-dependencies
shan skills move scope down tips,linear        # batch move
```

`scope up` moves from project to user. `scope down` moves from user to project. The skill's library directory is relocated and the outfit symlink is updated atomically.

Moves are dependency-aware. If a target depends on other skills and the move would break the graph, shan refuses by default. Pass `--cascade-dependencies` to expand the move set downward through the dependency closure. Shan never auto-moves dependents.

## Move skills between commitments

Convert a pluggable skill to core (or vice versa):

```sh
shan skills move commitment up playwright      # pluggable → core (copy to outfit)
shan skills move commitment down playwright    # core → pluggable (move to library)
```

`commitment up` copies the library skill into the outfit as a real directory — shan stops managing it. `commitment down` moves a core directory into the library and creates a symlink.

## Import skills from skills.sh

Use shan's importer when you want external skills to become first-class shan-managed skills instead of a separate install system:

```sh
shan skills install vercel-labs/agent-skills --skill typed-api-dx-review
shan skills install vercel-labs/agent-skills --skill playwright --skill linear
shan skills install vercel-labs/agent-skills --all --scope user
```

The importer is intentionally non-interactive: pass one or more `--skill` flags or `--all`.

Behavior:

- installs into the canonical Claude outfit first
- normalizes the imported skill names into shan's naming/layout
- converts the imported skills into pluggable library entries
- generates namespace routers when needed
- reconciles any configured agent mirrors afterward by re-symlinking whole agent skills directories

## Undo and redo

Every mutation records a snapshot. Undo restores the outfit to the state before the operation:

```sh
shan skills undo                               # undo last operation
shan skills undo 3                             # undo last 3 operations
shan skills redo                               # redo last undone operation
shan skills redo 2                             # redo last 2
shan skills undo --scope user                  # undo at user scope
```

Composite operations (like a batch `move scope down tips,linear`) undo as a single unit — all sub-actions reverse together.

## View history

```sh
shan skills history
shan skills history --scope user
```

Shows the operation log: what changed, when, and the targets involved.

## Run health checks

```sh
shan doctor skills                             # detect + auto-fix
shan doctor skills --no-fix                    # report only
```

Doctor runs 18 diagnostic aspects: agent-mirror, broken-symlink (with git rename detection), state-drift, new-leaf, stale-router, orphaned-router, orphaned-scope, stale-gitignore, corrupt-library-entry, frontmatter-mismatch, name-conflict, duplicate-name, shadow, stale-shadow, cross-scope-install, dependency-declaration, dependency-cycle, and dependency-active-graph.

Default mode detects and fixes. Use `--no-fix` to see findings without changes.

Dependency-specific doctor behavior:

- invalid dependency declarations are reported but not auto-rewritten
- active graph drift is auto-fixable when shan can legally activate the missing dependency closure
- namespace reach-in, cycles, malformed dependency fields, and prohibited cross-scope candidates stay report-only
